# 今日鸟签小程序 V0.1 内测版 PRD

更新时间：2026-06-05

目标上线内测时间：2026-06-07 周日

## 1. 背景

当前项目以 H5 页面承载“每日鸟签”玩法。2.0 版本计划迁移到微信小程序，围绕 NFC 手机支架建立“实物解锁 + 唯一小鸟宠物 + 每日鸟签 + AI 情绪陪伴 + 微信好友串门”的产品形态。

V0.1 不是完整商业版，目标是先跑通内测主链路：

1. 已购买用户可以登录并解锁自己的小鸟宠物。
2. 用户每天可以收到宠物叼来的鸟签。
3. 用户可以和宠物进行有限次数的文字情绪陪伴对话。
4. 用户可以通过微信分享宠物主页，好友点击后触发串门关系。
5. 运营侧可以手动发放兑换码、查询用户、处理异常解锁。

## 2. 已确认约束

| 事项 | 当前决策 |
|---|---|
| 小程序主体 | 企业主体 |
| 微信支付商户号 | 暂未开通 |
| 付费订阅 | V0.1 不做，先免费内测 |
| 老用户解锁 | 手动发放兑换码 |
| 新 NFC 一物一码 | 暂时不能立刻重写已售/现有 NFC |
| AI 方向 | 情绪陪伴为主，不做百科问答型助手 |
| 社交方向 | 只挖掘微信好友关系，小程序内不做私信、不做陌生人发现、不做玩家互联广场 |

## 3. 产品定位

今日鸟签小程序是一个由 NFC 手机支架唤醒的微信小鸟宠物系统。用户通过购买实物支架获得解锁资格，登录后拥有一只唯一小鸟。小鸟每天叼来一张鸟签，也可以用文字陪伴用户，并通过微信好友分享发生轻量串门。

## 5. 用户类型

| 用户类型 | 说明 | 权限 |
|---|---|---|
| 未登录访客 | 通过分享、H5 或小程序入口进入 | 只能看到引导登录/购买提示 |
| 未解锁用户 | 已登录，但无兑换码或 NFC 解锁资格 | 不能拥有宠物，看到购买引导 |
| 老支架用户 | 小程序上线前购买，NFC 跳 H5 | 使用兑换码解锁 |
| 内测用户 | 已兑换成功 | 拥有宠物、鸟签、AI 对话、分享串门 |
| 运营人员 | JOYI 后台使用者 | 管理兑换码、用户、宠物、AI 用量 |

## 6. 老支架定义

老支架指：小程序正式上线前，所有已售出且 NFC 跳转到 H5 的 NFC 手机支架。

老支架不再尝试通过 NFC 区分身份，统一使用兑换码解锁。

解锁关系：

```text
redeem_code -> user_id -> pet_id
```

## 7. 权限模型

V0.1 采用“用户拥有宠物，兑换码/NFC 只是解锁资格”的模型。

```text
user_id -> pet_id
redeem_code -> unlock_record -> user_id
tag_id -> unlock_record -> user_id
```

V0.1 因为暂时不能重写新 NFC，一物一码可以先作为后台结构预留，不作为主要验收项。

### 7.1 解锁规则

1. 用户必须先微信登录。
2. 用户未解锁时，进入锁定页。
3. 用户输入有效兑换码后，系统创建宠物。
4. 一个兑换码只能兑换一次。
5. 一个用户 V0.1 默认只能拥有一只宠物。
6. 用户解锁成功后，之后从小程序直接进入宠物主页。

### 7.2 无权限提示

未解锁用户进入时显示：

```text
你还没有唤醒自己的小鸟。
前往小红书「JOYI博物工作室」购买 NFC 手机支架，解锁专属鸟宠。
```

## 8. 核心用户流程

### 8.1 老用户兑换流程

1. 用户打开小程序。
2. 点击微信登录。
3. 系统判断用户未解锁。
4. 用户点击“我已购买，输入兑换码”。
5. 输入运营手动发放的兑换码。
6. 后端校验兑换码。
7. 创建 pet_id。
8. 生成宠物资料。
9. 进入宠物主页。

### 8.2 每日鸟签流程

1. 已解锁用户进入宠物主页。
2. 系统检查今天是否已生成鸟签。
3. 如果未生成，生成今日鸟签。
4. 如果已生成，展示今日固定鸟签。
5. 用户可以分享今日鸟签或宠物主页。

### 8.3 AI 情绪陪伴流程

1. 用户进入宠物主页。
2. 点击“和小鸟说话”。
3. 系统检查今日剩余免费次数。
4. 用户发送文字。
5. 后端拼接宠物人设、最近对话、用户输入。
6. 调用国内大模型。
7. 返回小鸟风格回复。
8. 消耗一次免费额度。
9. 次数用完后提示明天再来。

V0.1 次数建议：每日 5 轮。

### 8.4 微信好友串门流程

1. 用户在宠物主页点击分享。
2. 好友从微信聊天点击进入小程序。
3. 好友登录。
4. 如果好友无宠物，显示购买/解锁提示，同时可看到被分享宠物的公开信息。
5. 如果好友已有宠物，系统建立两只宠物之间的串门关系。
6. 被访问用户下次进入时，可以看到“有朋友的小鸟来串门了”。

串门只由微信分享触发。小程序内不提供陌生玩家列表、搜索玩家、私信或主动访问陌生宠物能力。

### 9.5 宠物主页

6. 串门提醒。

### 9.6 今日鸟签页

内容：

5. 分享按钮。

### 9.7 AI 聊天页

2. 输入文字。

V0.1 不做语音、不做图片、不做多轮复杂记忆。

### 9.8 串门结果页

好友点击分享后进入。

状态：

1. 好友未登录：提示登录。
2. 好友未解锁：展示购买/兑换引导。
3. 好友已解锁：展示两只小鸟见面结果。

## 10. 宠物生成规则

V0.1 目标是“唯一感”，不是复杂养成。

宠物资料包含：

```text
pet_id
owner_user_id
bird_id
pet_name
personality
tone
created_at
```

生成维度：

1. 鸟种：从现有鸟类数据中选择。
2. 性格：温柔、慢热、机灵、安静、嘴硬但关心人等。
3. 口吻：短句、轻陪伴、不说教。
4. 初始名字：系统可给默认名，用户可修改。

V0.1 可以使用确定性随机：

```text
hash(user_id + created_at) -> bird_id/personality/tone
```

## 11. AI 情绪陪伴需求

### 11.1 角色定位

小鸟不是客服，不是百科机器人，不是心理医生。它是用户通过 NFC 支架唤醒的小鸟宠物，主要提供轻陪伴、情绪安抚、日常回应。

### 11.2 回复原则

1. 简短。
2. 温柔。
3. 有小鸟角色感。
4. 不做医疗、法律、金融等高风险建议。
5. 不假装知道用户现实环境。
6. 不制造强依赖。

### 11.3 免费次数

V0.1 建议：

```text
每日免费 5 轮
每轮用户输入最多 300 字
每轮 AI 回复最多 200 字
```

### 11.4 记忆策略

V0.1 只保留短上下文：

1. 最近 6-10 条消息。
2. 宠物基础人设。
3. 不做长期记忆。

后续 V1.1 再做长期记忆和订阅权益。

## 12. 社交关系需求

### 12.1 原则

社交关系只通过微信好友分享触发。小程序内部不提供陌生人社交能力。

### 12.2 V0.1 支持

1. 分享宠物主页给微信好友。
2. 好友点击进入。
3. 双方都有宠物时，建立关系。
4. 记录串门事件。
5. 宠物主页显示最近串门提醒。

### 12.3 V0.1 不支持

1. 私信。
2. 用户搜索。
3. 陌生人推荐。
4. 世界频道。
5. 玩家排行榜。
6. 主动访问非微信分享来源的宠物。

## 17. 验收标准

V0.1 内测版验收时，需要跑通：

1. 新用户登录后看到锁定页。
2. 输入有效兑换码后成功创建宠物。
3. 同一个兑换码不能重复使用。
4. 已解锁用户再次进入直接看到宠物主页。
5. 用户可以查看今日鸟签，且同一天固定不变。
6. 用户可以进行 AI 文字对话。
7. 每日对话次数用完后不能继续发送。
8. 用户可以分享宠物主页给微信好友。
9. 好友已有宠物时，点击分享后建立串门关系。
10. 好友无宠物时，只展示引导，不建立关系。
11. 后台可以创建、查看、作废兑换码。
12. 后台可以查询用户和宠物。

## 21. 当前最大风险

1. 周日前时间很短，必须以“能跑通”为目标，不能追求完整商业版。
2. 微信登录、分享链路、审核和真机体验可能产生额外时间。
3. AI 接入可以加速开发，但不能替代产品边界控制。
4. 没有微信支付商户号，所以 V0.1 不能承诺订阅。
5. 新 NFC 暂时不能重写，所以 V0.1 主要依赖兑换码。

## 22. 决策结论

V0.1 内测版按以下原则推进：

1. 免费内测。
2. 老用户兑换码解锁。
3. 用户绑定唯一宠物。
4. AI 只做文字情绪陪伴。
5. 社交只做微信好友分享串门。
6. 小程序内不做私信和陌生人关系。
7. 周日目标是跑通链路，不是完整商业上线。
