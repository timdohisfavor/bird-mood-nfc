## 一、执行经验总结

### 1. H5 / UI 任务如果只看资产、不读运行态，容易做偏

导致问题的做法：

- 把项目当成“静态素材目录”而不是“运行中的 H5 产品”
- 只看文件，不打开本地预览
- 不消化浏览器批注，直接零碎改文案或样式

最终正确做法：

- 先确认项目资产、代码入口、运行中的 H5 结构
- 起本地服务后再看页面状态
- 优先从运行态理解：首页、抽签后状态、图鉴、分享页、NFC 路径
- 浏览器批注先收集，等你说“给完了”后再集中修改

教训：

- 这个项目不是“做一张网页”，而是一个状态驱动的 H5 产品
- 所有 UI 判断都应建立在实际运行效果上，而不是静态猜测上

### 2. 每日固定一签会干扰视觉调试，缺少重置入口会卡住迭代

导致问题的做法：

- 只保留真实产品态，不给本地预览一个“回到初始态”的入口
- 导致首页一旦抽过签，就很难继续改抽签前首屏

最终正确做法：

- 产品逻辑保留“每日固定一签”
- 但本地预览必须提供 reset / fallback / 可回退到初始状态的机制

教训：

- 真实产品逻辑和本地调试逻辑要分开设计
- 凡是有持久化状态的界面，开发时都要给自己留复位口

### 3. 大批量鸟图生成如果不先定风格，会返工

导致问题的做法：

- 一开始就全量生成 34 只
- 没先锁定风格标准
- 只输出图片，不同步到项目资产与 H5
- 保留背景、枝叶、场景，和现有项目资产不一致

最终正确做法：

- 先拿 4 只新增鸟做风格验证
- 再按每批 6 只推进
- 每批完成后回头检查风格一致性
- 最终要求严格统一：
  - 写实参考
  - 抓关键物种特征
  - 可爱圆润
  - 手绘质感
  - 非矢量
  - 独立鸟体
  - 透明底
  - 无树枝、叶子、场景、背景
  - `720x720`
- 产出后必须同步进项目实际资产目录和 H5 数据链路

教训：

- 这类资产任务不能只把“生成成功”当完成
- “可直接上线到项目里”才算完成

### 4. 参考图筛选如果不先做本地图库，效率很低

导致问题的做法：

- 单张单张搜
- 不保留筛选结果
- 不把用户补充的真实参考图纳入统一筛选面板

最终正确做法：

- 先生成本地参考图库页面
- 同步输出筛选 CSV
- 对 34 只鸟统一筛选“最具物种特征、最可爱、最适合封面”的图
- 用户手动给出的图优先保留

教训：

- 面对一组视觉对象时，先搭检视界面，再做判断
- “能看全局”比“逐条修”更高效

### 5. WebP / 构建链问题如果只盯最终报错，定位会慢

导致问题的做法：

- 直接认为构建失败就是资源错了
- 把旧的本地服务异常当成当前代码异常
- 每次 UI 改动都跑完整构建

最终正确做法：

- 区分“服务异常”和“资源/构建异常”
- 之前 `4173` 的 `502` 是旧服务问题，不是项目资源问题
- 用新端口启动干净服务验证，例如 `4183`
- 对 UI 高频迭代，增加快速构建路径
- 对正式发布，再跑完整 `npm run build`

教训：

- 慢不一定是错，先拆出“服务层 / 构建层 / 资源层”
- 本地高频迭代必须有快路径

### 6. Docker 化如果只追求理论完整，会拖慢交付

导致问题的做法：

- 为了“完全容器化的纯洁性”卡在不必要的问题上
- 把 helper service 和主应用构建混在一起硬拧
- 不做实机启动验证

最终正确做法：

- 目标定义成“本地一键启动并验证通”
- 同时交付：
  - `Dockerfile`
  - `docker-compose.yml`
  - `.env` 读取
  - 端口规划
  - README 启动命令
- 遇到真实问题直接修：
  - `magick` 不可用就改 `convert`
  - Postgres init mount 不稳就移除，改成应用自建表

教训：

- 本地基础设施任务的完成标准是“跑起来”
- 能用，比形式完整更重要

### 7. Git 提交如果不控范围，会把无关改动一并带走

导致问题的做法：

- 在脏工作区里直接全量提交
- 把 Docker / README / backend / bird sign 多条线混进一个提交

最终正确做法：

- 只提交当前任务范围内的文件
- 现有无关改动保留在工作区，不回滚、不混提

教训：

- 这个项目经常多线并行，提交必须按任务边界切

### 8. Netlify 发布如果每改一次就发一次，会浪费 credits

导致问题的做法：

- 把每次局部修改都当成一次部署节点
- 未做本地检查就直接上 Netlify

最终正确做法：

- 本地先完成一整批修改
- 先做：
  - 本地预览
  - `dist/` 检查
  - NFC 路径检查
  - 图鉴与首页检查
- 用户确认后再统一发布

教训：

- 这个项目的部署策略应以“合批发布”优先

### 9. 安装类任务如果只看“已安装”，结果常常不可靠

导致问题的做法：

- 只检查文件存在
- 不做 runtime probe
- 不做 live validation

最终正确做法：

- 安装完必须继续验证：
  - 命令是否可执行
  - 运行时是否正常
  - 实际目标功能是否可用

教训：

- “安装完成”不是结果
- “能执行并完成目标任务”才是结果

### 10. 长任务不报状态，用户会误以为卡死

导致问题的做法：

- 图片转换、WebP 重建、全量构建时不解释慢在哪里

最终正确做法：

- 明确说明正在运行的脚本
- 说明为什么慢
- 说明是否只是等待，不是死锁

教训：

- 长任务必须带进度语义，不然用户只能靠猜

## 二、你的偏好与理念提炼

### 1. UI 设计偏好

- 偏好“自然沉浸式”而不是通用 SaaS 卡片风
- 首页应该是体验入口，不是说明页
- 更重视情绪氛围、产品感和仪式感，而不是功能堆叠
- 插画 / 视觉资产必须有统一世界观，不能风格漂移
- 真实参考可以保留，但最终视觉不能太写实、太硬，也不能变成扁平矢量图
- 喜欢可爱圆润、手绘质感、带一点高级感和治愈感的画面
- 不能接受背景、枝叶、场景元素混入本应是“项目资产”的独立鸟体图
- 你会直接在浏览器上给具体批注，希望按视觉结果说话

### 2. 产品设计理念

- 第一期要轻，不要过早上账号体系和互动体系
- 核心不是“功能多”，而是“抽鸟签体验是否成立”
- 每日固定一签是产品核心感受之一
- 图鉴点亮、分享海报、NFC 唤起都服务于主体验，不应喧宾夺主
- 本地状态是可以接受的，只要体验闭环成立
- 更倾向先把单机体验做扎实，再考虑更重的系统化能力

### 3. 交互原则

- 抽签前、抽签后、第二天再来，这几个状态必须清楚
- 本地预览要有 fallback，不能让真实产品限制开发调试
- 图鉴必须有已收集 / 未收集视图，不然 34 只鸟很难浏览
- `/nfc/xxx` 这类真实入口，本地也必须能测
- 可见文案你会精修，希望助手等你一轮给完再统一改

### 4. 开发与协作偏好

- 偏好直接操作、直接验证，不喜欢只给理论方案
- 偏好先读上下文和现有资产，再动手
- 偏好分批推进、每批检查，不喜欢一次性大改后整体返工
- 偏好任务边界清晰，提交边界也清晰
- 偏好本地先稳定，再发 Netlify
- 对 credits、构建耗时、重复部署成本有明确意识
- 对“是否真的能用”非常敏感，重结果不重形式

## 三、可复用规则清单

以下规则为后续 Codex 会话默认遵循：

### A. 执行规则

1. 遇到这个项目的 H5 / UI 需求，先启动本地预览并阅读运行态，再开始修改。
2. 涉及浏览器批注时，先收集全部批注，等用户明确说可以统一修改后再批量改。
3. 涉及图片资产生成时，必须先做小批风格验证，再做全量。
4. 资产任务的完成标准是“已同步到项目资产与 H5”，不是“图片生成成功”。
5. 构建或预览异常时，先区分是旧服务问题、端口问题，还是代码/资源问题。
6. 高迭代阶段优先用快速构建路径；正式交付前再跑完整构建。
7. 安装类任务必须做 live validation，不能只汇报“已安装”。
8. 长耗时任务必须主动汇报当前脚本、慢的原因、是否仍在运行。
9. 脏工作区下只提交当前任务相关文件，不得顺手混提无关改动。
10. 未经用户确认，不要为了省事回滚现有非本任务改动。

### B. 产品规则

1. 第一期默认不引入账号系统和线上互动体系，除非用户明确改范围。
2. 首页默认服务于“抽签体验”，不是 landing page。
3. 每日固定一签是默认产品逻辑。
4. 本地预览默认应提供调试 fallback 或状态重置路径。
5. `/nfc/xxx` 路径默认应在本地可测。
6. 发布前默认检查 `dist/` 是否干净，不带 zip、源码脚本、后端目录、临时文件。
7. Netlify 默认采用“本地合批完成后再统一发布”的策略。

### C. 视觉规则

1. 新增鸟图默认遵循：
   - 写实参考
   - 抓关键物种特征
   - 可爱圆润
   - 手绘质感
   - 非矢量
   - 独立鸟体
   - 无背景
   - 无枝叶
   - 无场景
   - 透明底
   - `720x720`
2. 任何新资产都要检查是否与现有 34 只项目资产风格一致。
3. 参考图筛选默认输出本地图库页和结构化清单，便于用户审阅。
4. 用户亲自给出的参考图优先级高于自动检索结果。

### D. 沟通规则

1. 优先用中文沟通这个项目。
2. 回答以结果、路径、下一步为主，少讲空泛解释。
3. 当任务未完成但仍在推进时，要明确说明“还在运行”和原因。
4. 当用户关心成本时，显式说明构建成本、部署成本或迭代成本。
